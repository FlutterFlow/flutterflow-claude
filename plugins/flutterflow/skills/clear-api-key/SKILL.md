# Clear the FlutterFlow API key

The key can sit in several independent places: Claude Code's secure store, the
plugin's bridged env file, the CLI's credential cache, per-workspace `.env` files,
and possibly shell profiles or history. This skill locates and clears each, then
points at the one action that actually revokes the key.

## Read this first

- **Deleting local copies does NOT revoke the key.** The only way to truly kill a
  key is to rotate it server-side at <https://app.flutterflow.io/account>. If the
  key may have leaked (committed, synced, backed up, pasted somewhere), rotate it —
  everything below is local cleanup, not revocation.
- **Never delete the `Claude Code-credentials` or `Claude Safe Storage` keychain
  items.** Those are Claude Code's own storage, not the FlutterFlow token; removing
  them breaks Claude Code. The plugin token is not stored under a `flutterflow`-named
  keychain service, so there is no keychain item for you to delete by hand — use
  step 1 instead.
- **Discover before you destroy.** Run the read-only discovery in each step, show the
  user what was found, and get explicit confirmation before editing shell profiles or
  history, or deleting a workspace's `.env`. Prefer the targeted removals below over
  broad `rm`.

## 0. Preamble (puts the CLI on PATH for `flutterflow ai logout`)

```bash
export PATH="$HOME/.pub-cache/bin:/opt/homebrew/bin:/usr/local/bin:$PATH"
for d in "$HOME/flutter/bin" "$HOME/development/flutter/bin" "$HOME/fvm/default/bin" \
  "$HOME/.puro/envs/default/flutter/bin" /opt/flutter/bin /usr/local/flutter/bin; do
  [ -d "$d" ] && PATH="$d:$PATH"
done
export PATH
```

## 1. Claude Code secure store (the token you configured)

This is the token entered via `/plugin configure`. Claude Code keeps it in its own
secure store (OS keychain / encrypted "Safe Storage") — **not** in a plaintext JSON,
so clear it through Claude Code's UI rather than editing files or the keychain:

```text
/plugin configure flutterflow@flutterflow
```

Clear the API-token field and save, then **restart the session**. The SessionStart
hook only reconciles on a new session — on the next start, seeing no token, it
deletes the bridged env file from step 2 automatically.

## 2. Bridged plaintext env file

The hook writes the key to `~/.config/flutterflow/claude-env.sh` (exporting
`FF_API_KEY` and `FLUTTERFLOW_API_TOKEN`). Remove it directly instead of waiting for
a restart:

```bash
rm -f "$HOME/.config/flutterflow/claude-env.sh"
rmdir "$HOME/.config/flutterflow" 2>/dev/null || true   # only removes it if now empty
```

## 3. FlutterFlow CLI credential cache

`flutterflow ai init` / login caches the key in `~/.flutterflow/credentials.json`.
Use the CLI's `--all` flag — **bare `flutterflow ai logout` only *lists* saved keys
and clears nothing** (and exits 0) — then delete the file outright to be certain. It
is regenerated on the next `init`, and the CLI reads auth from the env, not this file:

```bash
flutterflow ai logout --all 2>/dev/null   # clears saved keys when the CLI is on PATH
rm -f "$HOME/.flutterflow/credentials.json"   # definitive; also covers the CLI-absent case
```

## 4. Per-workspace env files

Every `flutterflow ai init` writes `<workspace>/.flutterflow/.env` containing
`FF_API_KEY` (and `FF_DSL_PROJECT_ID`). There can be **many**. First list them —
this prints absolute, space-safe paths:

```bash
find "$HOME" -type d -name .flutterflow \
  -not -path '*/Library/*' -not -path '*/node_modules/*' -not -path '*/.git/*' 2>/dev/null \
  | while read -r d; do [ -f "$d/.env" ] && echo "$d/.env"; done
```

Show the list and get the user's OK. Then delete each `.env` — it's regenerated by
`flutterflow ai init --resume`. This loop reuses the exact paths from the search, so
there's no manual `<workspace>` substitution to get wrong (no doubled paths, no
unexpanded `~`, spaces handled):

```bash
find "$HOME" -type d -name .flutterflow \
  -not -path '*/Library/*' -not -path '*/node_modules/*' -not -path '*/.git/*' 2>/dev/null \
  | while read -r d; do [ -f "$d/.env" ] && rm -f "$d/.env" && echo "removed: $d/.env"; done
```

To keep other (non-secret) vars in a specific file instead of deleting it, strip only
the secret lines. Operate on the **absolute path** printed above (never a `~`-form),
and preserve the file's permissions by rewriting in place:

```bash
f="/absolute/path/.flutterflow/.env"; tmp="$(mktemp)" \
  && grep -vE '^(FF_API_KEY|FLUTTERFLOW_API_TOKEN)=' "$f" > "$tmp" \
  && cat "$tmp" > "$f" && rm -f "$tmp"
```

Also scan for stray `.env` files where the key may have been pasted by hand, outside
any `.flutterflow` dir (widen the roots as needed):

```bash
grep -rlE '^(FF_API_KEY|FLUTTERFLOW_API_TOKEN)=' \
  "$HOME"/Documents "$HOME"/Projects "$HOME"/dev "$HOME"/code "$HOME"/src . \
  --include='.env' --include='.env.*' --exclude-dir=node_modules --exclude-dir=.git 2>/dev/null || true
```

Show matches and confirm before editing. **If the key was ever committed to a git
repo, deleting the working copy does not remove it from history** — treat it as leaked
and rotate (see top of file).

## 5. Shell profiles (only if a manual export was added)

Scan; do **not** auto-edit. Show the matching lines and confirm before removing:

```bash
grep -nE 'FF_API_KEY|FLUTTERFLOW_API_TOKEN' \
  "$HOME"/.zshrc "$HOME"/.zprofile "$HOME"/.zshenv \
  "$HOME"/.bashrc "$HOME"/.bash_profile "$HOME"/.profile \
  "$HOME"/.config/fish/config.fish 2>/dev/null
```

If a line is found, back up the file (`cp file file.bak`) and delete only that line
with the user's OK — then `rm -f file.bak` once it looks right, since the backup still
contains the key.

## 6. Shell history

If the key was ever typed on a command line (e.g. a manual `export` or `--api-key`),
it is in history. Report matches first — counts only, never print the value:

```bash
for h in "$HOME/.zsh_history" "$HOME/.bash_history"; do
  [ -f "$h" ] && printf '%s: %s match(es)\n' "$h" \
    "$(grep -cE 'FF_API_KEY|FLUTTERFLOW_API_TOKEN|--api-key' "$h" 2>/dev/null)"
done
```

Only if matches exist and the user confirms — and after warning them to close other
terminal windows first (a running shell can rewrite history on exit and undo this) —
scrub, reading from the backup and writing the real file (safe against truncation):

```bash
cp "$HOME/.zsh_history" "$HOME/.zsh_history.bak" \
  && grep -vE 'FF_API_KEY|FLUTTERFLOW_API_TOKEN|--api-key' "$HOME/.zsh_history.bak" > "$HOME/.zsh_history"
```

Then confirm `~/.zsh_history` still looks intact and **delete the backup — it holds
the unredacted key** (leaving it would just relocate the leak):

```bash
rm -f "$HOME/.zsh_history.bak"
```

## 7. Current shell environment

Exported vars in already-running shells can't be cleared from here. Tell the user to
run this in each open terminal:

```bash
unset FF_API_KEY FLUTTERFLOW_API_TOKEN
```

## 8. Verify

Confirm every local copy is gone — including a re-scan of workspace `.env` files
(the earlier steps are per-file confirmed, so this catches any you skipped):

```bash
ls "$HOME/.config/flutterflow/claude-env.sh" 2>/dev/null && echo "bridge file: STILL PRESENT" || echo "bridge file: cleared"
ls "$HOME/.flutterflow/credentials.json"     2>/dev/null && echo "CLI cache: STILL PRESENT"   || echo "CLI cache: cleared"
echo "workspace .env files still holding the key (none = clean):"
find "$HOME" -type d -name .flutterflow \
  -not -path '*/Library/*' -not -path '*/node_modules/*' -not -path '*/.git/*' 2>/dev/null \
  | while read -r d; do [ -f "$d/.env" ] && grep -lE '^(FF_API_KEY|FLUTTERFLOW_API_TOKEN)=' "$d/.env"; done \
  | sed 's/^/  /'
```

Two things this file check can't prove:

- The **secure-store token** (step 1) is only confirmable by reopening
  `/plugin configure flutterflow@flutterflow` and seeing the field empty.
- The **authoritative** proof the key is dead is that `flutterflow ai status
  <project-id>` fails authentication — and that only holds once you've rotated the key
  server-side. Local deletion alone does not make `status` fail.

## What this cannot reach

- **The key on FlutterFlow's servers** — only rotating it at
  <https://app.flutterflow.io/account> invalidates it. Do this if it may have leaked.
- **Backups & sync** — Time Machine snapshots, iCloud/Dropbox copies of `~/.config`
  or a synced workspace, and editor swap/undo files may still hold a plaintext copy.
  If the key was ever in a synced or backed-up plaintext file, treat it as exposed
  and rotate.
- **Other machines and CI** — anywhere else you set `FF_API_KEY`: other laptops, CI
  secrets, `.env` files in other repos.
